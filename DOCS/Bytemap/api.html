Module bytemap

Byte-flavored memory bitmap, useful for loading image-style assets from byte streams (and vice versa), where components are integers ranging from 0 to 255.

While bytemaps can be used like Corona's memory bitmap, the latter should be preferred for per-pixel operations. More typical bytemap usage will involve bulk operations, such as reading entire images from custom file formats or network downloads.

To use the plugin, add the following in build.settings:

plugins = {
   ["plugin.Bytemap"] = { publisherId = "com.xibalbastudios" }
 }

Sample code is available here.

The Bytes type—specified in a few of the bytemap methods—may be any object that implements ByteReader, including strings.

Creates a new Bytemap.

Bytemaps are external textures and thus integrate with Corona's display objects via their filename and baseDir properties, e.g. in paints, as arguments to display.newImage, etc.

Bytemap:GetBytes ([opts])
Gets a bytemap's bytes.

More specifically, gets the region from (x1, y1) in the upper-left to (x2, y2) in the lower-right. By default, these have values (1, 1) and (width, height) respectively (cf. newTexture), meaning the full bytemap.

Coordinates will be clipped if necessary to stay within this full region. Furthermore, they are sorted so that x1 < x2 and y1 < y2.

If two related coordinates are outside in the same way—x1 and x2 both < 1, for instance—the region is empty. In this case, a zero-length string is returned.

Parameters:

  • opts table Options for getting bytes, which include:

    • get_info: If true, a table is populated with some final results (described below). The value may be a table, in which case it will be the one used; otherwise, a new table is created.
    • x1, y1, x2, y2: Custom region coordinates, cf. the summary.
    • format: One of "mask", "rgb", "rgba", "red", "green", "blue", "alpha", describing the format of the final bytes.

    When absent, the bytemap's contents are returned as-is. Otherwise, with any of the first three choices, the result is converted, lopping off or adding padding bytes as needed (for blue and green channels, using 0 bytes; the alpha channel is padded with a 255 byte).

    The remaining options emit a result in one-byte-per-pixel form, containing the bytes from each pixel belonging to the named component. If the bytemap has no such channel (e.g. its format is "mask", but "green" or "alpha" was requested), a zero-length string is returned. ("red" and "mask" are synonyms.)

Returns:

  1. string A copy of the requested bytes.
  2. optional table If get_info was provided as an option, the info table.

    The final region coordinates will be added under the x1, y1, x2, and y2 fields. In the case of empty regions these will be the original values (though sorted).

    Additionally, a format field will be assigned one of "mask", "rgb", or "rgba", describing the underlying format of the returned bytes. In particular, if anything other than "rgb" or "rgba" was specified as the format option, this will be "mask".

    For non-empty regions, the returned bytes are to be understood as a (x2 - x1 + 1) × (y2 - y1 + 1)-sized rectangle with bytes-per-pixel corresponding to format.

Bytemap:SetBytes (bytes[, opts])
Sets a bytemap's bytes.

The region details largely carry over from Bytemap:GetBytes, although see the note about custom region coordinates below. No bytes change when the region is empty.

If too few bytes are provided to service the requested region, it will be clipped against the last line containing pixels. (Color-keyed pixels are considered present for purposes of this analysis.) Furthermore, any incomplete final pixel is discarded.

Parameters:

  • bytes Bytes Bytes to assign.
  • opts table Options for setting bytes, which include:

    • get_info: As per GetBytes.
    • x1, y1, x2, y2: Custom region coordinates, cf. the summary. bytes is interpreted as a (x2 - x1 + 1) × (y2 - y1 + 1)-sized rectangle. N.B. bytes sees the coordinates' unclipped values (though sorted), whereas an info table will contain them in their final form.
    • colorkey: If provided, a Bytes value of one or more bytes (zero-padded if necessary to agree with format; excess bytes are dropped). Pixels in bytes that match colorkey are skipped, leaving the bytemap contents intact.
    • format: One of "mask", "rgb", "rgba", "red", "green", "blue", "alpha", "grayscale", describing the format of bytes.

    If absent, the bytemap's format is used. Otherwise, with any of the first three choices, discrepancies are handled as follows: if bytes has more components than the bytemap, the extra bytes are not written; with fewer, the additional channels in the bytemap are left alone.

    The remaining options assume bytes to be in one-byte-per-pixel form. In the case of "grayscale", the byte is copied to each non-alpha channel of the bytemap's pixel. Otherwise, the byte is written to the named channel in that pixel (this is a no-op if the bytemap has no such channel, cf. the similar note for Bytemap:GetBytes).

Returns:

    optional table The info table, if requested.

    The table contents match those for Bytemap:GetBytes, except now they describe the assigned—rather than returned—bytes.
