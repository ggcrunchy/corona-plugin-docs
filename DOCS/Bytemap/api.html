Module bytemap

Byte-flavored memory bitmap, useful for loading image-style assets from byte streams (and vice versa).

To use the plugin, add the following in build.settings:

plugins = {
   ["plugin.Bytemap"] = { publisherId = "com.xibalbastudios" }
 }

Sample code is available (TODO).

The Bytes type—specified in a few of the bytemap methods—may be any object that implements ByteReader, including strings.

newTexture (config)
Creates a new Bytemap.

Bytemaps are external textures and thus integrate with Corona's display objects via their filename and baseDir properties, e.g. in paints, as arguments to display.newImage, etc.

Parameters:

  • config table

    Parameters used to initialize the bytemap:

    • width: Number of "pixels" wide...
    • height: ...and tall.
    • format: Underlying data format, which may be "mask", "rgb", or "rgba".

    Bytemap The new bytemap.
Bytemap:GetBytes ([opts])
Gets a bytemap's bytes.

More specifically, gets the region from (x1, y1) in the upper-left to (x2, y2) in the lower-right. By default, these have values (1, 1) and (width, height) respectively, cf. newTexture, meaning the full bytemap.

Coordinates outside this full region will be clipped against it. Furthermore, they are sorted so that x1 < x2 and y1 < y2.

If two related coordinates are outside in the same way—both x1 and x2 < 1, for instance—we have an empty region, so a zero-length string will be returned.

Parameters:

  • opts table Get options, which include:

    • get_info: If true, a table will be populated with some final results of the get operation. If this also happens to be a table, this will be what is returned; otherwise, a new table is created.
    • x1, y1, x2, y2: Custom region coordinates, cf. the summary.
    • format: One of "mask", "rgb", "rgba", "red", "green", "blue", "alpha", describing the format of the final bytes.

    If absent, the bytemap's contents are returned as-is. Otherwise, with any of the first three choices, the result is converted, lopping off bytes or padding as needed (for blue and green channels, using 0 bytes; the alpha channel is padded with a 255 byte).

    The remaining options provide a result in one-byte-per-pixel form, containing the byte from each pixel that belongs to the named component. If the bytemap has no such channel (e.g. its format is "mask", and "green" or "alpha" was requested), a zero-length string is returned.

Returns:

  1. string Bytes.
  2. optional table If get_info was provided as an option, the info table.

    The final region coordinates will be added under the x1, y1, x2, and y2 fields. In the case of null regions these will be the original values (though sorted).

    Additionally, a format field will be assigned one of "mask", "rgb", or "rgba", describing the underlying format of the returned bytes. In particular, if anything other than "rgb" or "rgba" was specified in the format option, this will be "mask".

    For non-null regions, the returned bytes are to be understood as a (x2 - x1 + 1) x (y2 - y1 + 1)-sized region with bytes-per_pixel corresponding to format.

Bytemap:SetBytes (bytes[, opts])
Sets a bytemap's bytes.

Most of the region details carry over from Bytemap:GetBytes, except null regions are merely a no-op, rather than affecting a return value.

Parameters:

  • bytes Bytes Bytes to assign.
  • opts table Set options, which include:

    • get_info: As per GetBytes.
    • x1, y1, x2, y2: Custom region coordinates, cf. the summary. The bytes data is interpreted as having dimensions (x2 - x1 + 1) x (y2 - y1 + 1), with sorting but no clipping.
    • colorkey: If provided, a Bytes value of one or more bytes (zero-padded if necessary to agree with format; excess bytes are dropped). Pixels in bytes that match colorkey are skipped, leaving the bytemap contents intact.
    • format: One of "mask", "rgb", "rgba", "red", "green", "blue", "alpha", "grayscale", describing the format of bytes.

    If absent, the bytemap's format is used. Otherwise, with any of the first three choices, discrepancies are handled as follows: if bytes has more components than the bytemap, those are left unwritten; if it has fewer, on the other hand, the additional channels in the bytemap are left intact.

    The remaining options assume bytes to be in one-byte-per-pixel form. In the case of "grayscale", the byte is copied to all channels of the bytemap's pixel. Otherwise, the byte is written to the named channel in that pixel (this is a no-op if the bytemap has no such channel, cf. the similar note for Bytemap:GetBytes).

Returns:

    optional table The info table, if requested.

    The format field is much like that of Bytemap:GetBytes, except describing the assigned—rather than returned—bytes.

    However, in the case of non-null regions, the coordinates differ from Bytemap:GetBytes, describing the (x2 - x1 + 1) x (y2 - y1 + 1)-sized region that was actually visited (n.b. not necessarily written, due to color keys or insufficient bytes).
